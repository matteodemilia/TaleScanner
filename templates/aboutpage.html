    Overview

  • What is Tale Scanner?

    The Tale Scanner system is a natural language processor designed with the intention of providing a consistent alternative to inaccurate generalize artifical intelligence services. Tale Scanner can and should be utilized as a relatvie source for linguistic improvement. As all measurements of Tale Scanner are algorithmically determined, there may exist exceptions, anomalies, and unique error cases that have not been identified and/or handled yet.

    Tale Scanner's Goal

    The goal of the Tale Scanner system is to provide a consistent, reliable, and informative tool to conduct analysis and help speech language pathologists assess linguistic capability. By providing an algorithmic process based on pattern identification, the Tale Scanner system eclipses the consistency of generalized machine learning models, and often the accuracy as well.

    The user interface of the system is designed with simplicity and intuitiveness in mind. Tale Scanner notifies the user of time complexity warnings, labels all functionalities, and outlines further detail in both the results output, and this "About" page.

    Acknowledgement of Anomalies, Paramaters, and Analysis Failures

    Tale Scanner was developed by a collegiate team of four during a set time frame. Due to the nature of development, it is important to note that Tale Scanner is not a perfect analysis tool by any measure; and due to the nature of linguistic analysis, it never will be. Although some analysis choices may display entirely accurate results with high consistency, the process will still have errors if such a case were to arise.

    • How to Use Tale Scanner

  • Assumption of Practicality

    Although the Tale Scanner system is designed to handle as many special cases as possible, there exist many exceptions that will cause issue with a segment of the analysis process. Because of this, the user is expected to input bodies of text that are linguistically practical for writing standards. The more one includes impractical cases of writing, the more one should expect errors in the analysis results.

    Example of practical writing: [ As the sun set, the old man sat by the river, reminiscing about the days gone by. With a contented sigh, he closed his eyes, feeling grateful for a life well-lived. ]

    Example of impractical writing: [ The player named Gh0stk!d44.00 said: "Dude wth, I'm such a d****ss for paying 55.23gold..." ]

    Even if impractical cases are included in the text, Tale Scanner's output should be reliable from test to test. Thus, users can include special cases if the goal is to improve the same text through multiple iterations.

    The Homepage

    The Tale Scanner homepage acts as both the system's landing page and dashboard. All navigation options are accessible from the homepage, as well as all analysis functionalities. The homepage includes analysis choice-boxes, analysis input text-box, quality of life buttons (select all, deselect all, etc.), and a navigation headbar.

    User Input Choices

    The first relevant analysis functionality on the homepage is the list of analysis choice-boxes. The user can click any combination of these choice-boxes, and receive the correlated results in the following analysis results page. Currently, Tale Scanner can perform analysis on ten different types of linguistic analysis:

    1. Total word count of the text
    2. Total unique word count of the text
    3. Type token ratio of the text
    4. Total morphemes within the text
    5. Total number of clauses within the text
    6. Total number of subordinate clauses within the text
    7. Syntactic subordination index of the text
    8. Total number of verb errors within the text
    9. Verb error ratio of the text
    10. Average words per clause within the text

    Three of the analysis choices take significantly longer to analyze than the remaining seven choices. The longest analysis time is for analyzing morphemes, with an average of [1.4 seconds] per [10 words] in the text. Verb errors and verb error ratio have essentially the same processing time, as both require use of the same algorithm: averaging at approximately [1 second] per [10 words] in the text. The remaining seven analysis choices take [<1 second] to process any number of words (>600 words has not been tested extensively for time)


    By returning to the homepage via the navigation headbar (Tale Scanner logo), the user can retain all submission choices from the previous run and utilize them instantaneously for any future run(s).

    User Input Text-Box

    Just under the analysis choice-boxes is the input text-box which will receive the body of text that will be analyzed by the user and the Tale Scanner system. To submit your body of text for analysis, simply click on the [Submit] button. Assuming analysis choces have been made and text has been inputted, Tale Scanner will receive said choices and text, and start the analysis process. While the process is underway, the submit button will display a rotating loading bar as an indicator the process is functioning properly. However, just because the bar displays a loading state, does not necessarily mean Tale Scanner is processing.

    If users were to return to the input screen via the browser-window back arrow, the submitted body of text will remain in the input box, and the loading bar may still be rotating. In this case the submit button is still accessible, and can be clicked for another submission. Because of this, it should be a rule-of-thumb for users to attempt to use the button even if it displays the processing animation. As long as the submission correlating to the animation has been performed, the Tale Scanner process should be ready for more inputs.

    • Understanding Tale Scanner's Analysis

  • Foundational Processes

    In natural language processing, there are many computational processes that are utilized across a wide variety of analysis algorithms. Because Tale Scanner uses a handful of these processes in multiple algorithms, it is important to understand the processes themselves to avoid error parameters.

    Parsing

    Parsing is the process of splitting a segment of information into smaller logical syntactic components in which meaning is directly conveyed. A body of text can be parsed for keywords, sentences can be parsed by structure, and words can be parsed into smaller segments of meaning. Depending on the process, Tale Scanner utilizes parsing to either prime the algorithm with data, or to finalize calculations before counting and outputting.

    Segmentation and Tokenization

    Similarly to the parsing process, segmenting and tokenization are processes that split a segment of information into smaller components. However, these components do not need to convey meaning in themselves, as these processes are typically used for data isolation only. Segmenting is the act of splitting the data, and tokenization is the act of isolating the split data into tokens. In Tale Scanner, the body of text is segmented by words, creating a list of word tokens. The segmentation process uses characters to determine the beginning and end of a token. Currently, tokens are split by:

    1. White-spaces (' ')
    2. Most symbols (@, *, %, $, &, etc.)
    3. Some number formats (11:45, 50.45, 1/2, etc.)

    Tale Scanner Analysis

    In order to provide users with all of the tools necessary to conduct successful analysis, Tale Scanner highlights process details for each of the analysis types. When conducting analysis through Tale Scanner, the user will have access to dynamic analysis information via the details drop-downs. To provide more information on the analysis processes and to identify possible parameter errors, please refer to the following sections for specific analysis process information.

    Word Counts and Type Token Ratio

    To determine the total word count and total unique word count, Tale Scanner utilizes the spaCy natural language processing library. Once the user submits the necessary information, the body of text is processed through the spaCy pipeline, where segmenting and tokenization occurs. When segmenting the text, words are tokenized by the identification of "barriers" between the words. These barriers can take the form of white spaces (' '), varying punctuation ('!', '?', '.', etc), and varying symbols.

    Error parameters for total word count:

    1. [Character alternatives]: replacement characters break segmentation (e.g. "repl@c3ment" = "repl" "c" "3" "ment" = four words)
    2. [Symbols]: symbols are often a segment barrier (e.g. "smith@gmail.com" = "smith" "gmail" "com" = three words)
    3. [Non-English language]: other languages follow different rules, and can use different characters (e.g. "いただきます" has unidentifiable characters = zero words)
    4. [Excess numbers]: numbers count as words, but some formats create barriers (e.g. "12:35" = "12" "35" = two words)

    Once the body of text has been segmented and tokenized, it is placed within a list containing all generated word tokens. To determine total unique word count, a python function is utilized to create a new list containing only unique word tokens. The unique word list is then counted just as the total word count is. Because of this, the unique word count is affected by the same error parameters as total word count. As long as the body of text avoids said error parameters, the unique word count will provide the most conistently accurate result of all Tale Scanner analysis processes.

    The type token ratio(TTR) is dependent on the outputs of the total word and total unique word count functions. TTR is calculated by dividing the total unique word count, by the total word count. Assuming word count parameter barriers are avoided, the TTR will always provide an accurate result.
